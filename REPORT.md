## 1. Постановка сценария

Опишите сценарий "запрос на оплату -> обрыв сети -> повторный запрос".

Нужно отразить:
1. Что происходит без защиты.
2. Почему возможна повторная обработка одного и того же намерения клиента.

**Сценарий:** Клиент отправляет запрос на оплату заказа, но до получения ответа от сервера происходит обрыв сети. Клиент, не получив подтверждения, повторяет запрос. 

**Без защиты:** При использовании `mode='unsafe'` (без блокировок) запросы обрабатываются параллельно, и оба успешно изменяют статус заказа на "paid", создавая две записи в `order_status_history`. Это происходит потому, что нет механизма,识别вающего, что второй запрос является дубликатом первого.

**С Idempotency-Key:** При наличии заголовка `Idempotency-Key` middleware кэширует первый ответ и при повторном запросе возвращает его без повторной обработки. Оплата происходит только один раз.

## 2. Реализация таблицы idempotency_keys

Приведите итоговый DDL таблицы и краткое объяснение колонок.

Укажите:
- какие поля используются для идентификации запроса;
- как хранится кэш ответа;
- какие ограничения/индексы добавлены и зачем.

```sql
CREATE TABLE idempotency_keys (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    idempotency_key VARCHAR(255) NOT NULL,
    request_method VARCHAR(10) NOT NULL,
    request_path VARCHAR(512) NOT NULL,
    request_hash VARCHAR(64) NOT NULL,
    status VARCHAR(20) NOT NULL DEFAULT 'processing',
    status_code INTEGER,
    response_body TEXT,
    expires_at TIMESTAMP NOT NULL,
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW()
);
```

**Колонки:**
- `idempotency_key` — уникальный ключ, переданный клиентом в заголовке `Idempotency-Key`
- `request_method` + `request_path` — вместе с ключом идентифицируют конкретный endpoint
- `request_hash` — SHA256 хэш тела запроса для детектирования изменения payload
- `status` — состояние: 'processing' (выполняется), 'completed' (завершено), 'failed' (ошибка)
- `status_code` — HTTP-код ответа первого запроса
- `response_body` — тело ответа в JSON (сериализованная строка)
- `expires_at` — время истечения записи (TTL, по умолчанию 24 часа)

**Индексы:**
```sql
CREATE UNIQUE INDEX idx_idempotency_unique 
    ON idempotency_keys (idempotency_key, request_method, request_path);
```

Уникальный индекс обеспечивает защиту от конкурентных запросов с одинаковым ключом.

## 3. Реализация middleware

Опишите алгоритм middleware по шагам.

Минимум:
1. Чтение `Idempotency-Key`.
2. Проверка существующей записи.
3. Обработка кейса "тот же key + другой payload".
4. Сохранение результата первого запроса.
5. Возврат кэшированного ответа при повторе.

**Алгоритм middleware (IdempotencyMiddleware):**

1. **Фильтрация запросов:** Пропускаем только POST-запросы к `/api/payments/*`

2. **Чтение заголовка:** Извлекаем `Idempotency-Key` из headers. Если отсутствует — пропускаем запрос без обработки.

3. **Вычисление хэша:** Вычисляем SHA256 от тела запроса (`request_hash`).

4. **Восстановление тела запроса:** После чтения тела для передачи в `call_next` создаём собственную функцию `receive()`.

5. **Проверка существующей записи:**
   - Выполняем SELECT по ключу (idempotency_key, method, path)
   - Если запись найдена:
     - Сравниваем хэши. При несовпадении → **409 Conflict** ("Idempotency-Key conflict: same key with different payload")
     - Если статус 'completed' → возвращаем кэшированный ответ + заголовок `X-Idempotency-Replayed: true`
     - Если статус 'processing' → **409 Conflict** ("Request is already processing")

6. **Создание записи:** Если записи нет — создаём новую со статусом 'processing' и expires_at (текущее время + TTL)

7. **Обработка IntegrityError:** При конкурентной вставке (уникальный индекс) — повторно читаем запись и обрабатываем как в п.5

8. **Выполнение запроса:** Вызываем `call_next(request)` для обработки запроса

9. **Сохранение результата:** После выполнения обновляем запись: статус='completed', status_code, response_body

10. **Возврат ответа:** Возвращаем оригинальный ответ клиенту

## 7. Сравнение с решением из ЛР2 (FOR UPDATE)

Сравните подходы по сути и по UX.

Рекомендуется сравнить:
1. Цель механизма.
2. Что происходит при повторе запроса.
3. Где гарантия обеспечивается (API-слой vs БД-слой).
4. Нужно ли использовать оба механизма вместе.

**Запуск:** `pytest app/tests/test_compare_idempotency_vs_for_update.py -v -s`

| Аспект | FOR UPDATE (lab_02) | Idempotency-Key (lab_04) |
|--------|---------------------|-------------------------|
| **Цель** | Защита от race condition (конкурентные запросы от разных клиентов) | Защита от retry (повторный запрос того же клиента) |
| **Как работает** | Блокировка строки в БД через `SELECT ... FOR UPDATE` | Кэширование ответа по ключу в таблице |
| **Повтор запроса (последовательный)** | Возвращает business error: `"Order ... is already paid"` | Возвращает кэшированный успешный ответ |
| **Повтор запроса (параллельный)** | Второй запрос блокируется, затем получает error | Один из запросов может получить 409 "already processing" |
| **Уровень гарантии** | База данных (SQL) | Приложение (middleware) |
| **Ошибка при конфликте** | Бизнес-логика: "already paid" | HTTP: 409 Conflict |

**Вывод:** Оба механизма **не взаимоисключающие**, они решают **разные проблемы**:

1. **FOR UPDATE** не защищает от клиентского retry — клиент может повторить запрос, и бизнес-логика вернёт "already paid", но это уже повторный вызов, лишняя нагрузка.

2. **Idempotency-Key** не защищает от race condition — два разных клиента с разными ключами могут одновременно заплатить за один заказ.

**Рекомендация:** Использовать **оба механизма вместе**:
- Idempotency-Key обрабатывает retry на уровне API
- FOR UPDATE обрабатывает race condition на уровне БД

## 8. Выводы

Сформулируйте 3–5 практических выводов.

1. **Idempotency-Key — стандартный механизм защиты от дублирования.** Заголовок `Idempotency-Key` широко используется в платёжных API (Stripe, PayPal) именно для решения проблемы "клиент не получил ответ → повторил запрос".

2. **Кэширование на уровне приложения, а не БД.** В отличие от FOR UPDATE, middleware кэширует уже готовый HTTP-ответ, а не состояние БД. Это позволяет вернуть точную копию первого ответа (включая message, status_code).

3. **Обязательна проверка payload.** Просто сохранение по ключу недостаточно — нужно проверять, что payload не изменился (через хэш). Иначе клиент может использовать старый ключ с новыми данными.

4. **409 Conflict vs 200 OK.** Повтор с тем же ключом и payload → 200 (кэш). Повтор с тем же ключом, но другой payload → 409. Это важно для клиента — он должен понять, что что-то пошло не так.

5. **TTL необходим.** Записи в `idempotency_keys` должны истекать (по умолчанию 24 часа), иначе таблица будет расти бесконечно. Клиент может использовать ключ повторно через сутки — это нормально.

6. **Два уровня защиты = полная надёжность.** FOR UPDATE + Idempotency-Key — это best practice для платёжных систем. Первый уровень (БД) защищает от конкурентности, второй (API) — от сетевых сбоев.
